What lives where

stringeex_service.py
All HTTP calls to the StringeeX API. Handles multi-portal auth, pagination, and cross-portal deduplication. Nothing else touches the API.
sync_calls.py
The management command. Orchestrates fetch → filter → enrich → upsert. Can run one-shot, backfill, daemon, or from CSV.
team_lookup.py
Reads Team_tradex.xlsx once into memory. All enrichment (TL, pool, full name, employee code) happens here via prefix matching.
views.py
Django API endpoints. Queries the DB (never the API). Converts UTC ms timestamps to IST for hour-bucketing. Read-only — no writes.

The daemon's overlap trick

Why 9 minutes of data every 4 minutes? The overlap is intentional. The daemon fetches interval + 5 minutes on every cycle. This prevents gaps if a poll is slightly late. It's safe because update_or_create handles duplicate inserts.

Smart upsert — the two modes

today's records

Uses update_or_create. Always writes the latest data. This keeps live call data current throughout the day.

historical records

Uses get_or_create. Never overwrites. Once a day's data is stored, it's permanent — safe for audits and reports.

Why the API always ignores your date filter

The StringeeX API ignores start_time / end_time parameters entirely. It always returns records sorted newest-first. The code works around this by paginating until it hits records older than the requested start boundary, then stopping. This is why syncing a short window can still fetch many pages.

Backfill flow — step by step

1
Optionally flush: sync_calls --flush wipes all records. Do this before a clean backfill.
2
Run backfill: --start YYYY-MM-DD --end YYYY-MM-DD. Iterates day-by-day. Historical days use get_or_create; if today is in range, it uses update_or_create.
3
Start daemon: takes over from where the backfill left off, polling every 4 minutes.

CAN_NOT_MAKE_CALL — why counts don't match the export

Records with end_call_reason = 'CAN_NOT_MAKE_CALL' are filtered out at ingest time — both from CSV import and the API sync. They never enter the DB. If your call counts don't match the raw StringeeX export, this is the first thing to check.

How enrichment works

When a call comes in with account_name = "irfan@bluewave.com", the code strips everything after @ to get the prefix "irfan". That prefix is looked up in an in-memory dict built from Team_tradex.xlsx.

This is why irfan@bluewave.com and irfan@nexora.live match the same record — the domain is irrelevant.

Build order — Sheet2 beats Sheet1 silently

1
Sheet1 is loaded first. Keyed on the Email column (nexora.live addresses). Covers the full CRE list.
2
Sheet2 Stringee rows are loaded second and overwrite Sheet1 for any matching prefix. Keyed on Dialer Name (bluewave.com addresses). No error is raised — Sheet2 just wins.
Debugging agent TL/pool issues: if an agent appears in both sheets and their TL looks wrong, Sheet2 is authoritative. Check what prefix the API sends vs what's in Sheet2's "Dialer Name" column.

The two-step cache reload

After editing Team_tradex.xlsx, you must do both of these — doing only one leaves the other stale:

step 1 — fix the DB
python manage.py reenrich_teams

Re-reads the xlsx and bulk-updates every CallRecord row in the DB with fresh TL/pool/name data.
step 2 — fix memory
POST /api/reload-team/

Forces reload_team_lookup() in the running web process. Without this, the in-memory cache still holds the old data.

Things that look like bugs but aren't

talk_duration = 0
This is not a data error. It means the call was never answered. is_connected is simply talk_duration > 0. The distinction matters for connect-rate calculations.
timestamp confusion

start_time_ms in the DB is always raw UTC milliseconds (e.g. 1746700000000). To read it:

datetime.fromtimestamp(1746700000000 / 1000) → gives UTC datetime.

The dashboard and views.py convert to IST (UTC+5:30) for hour-bucketing. All date boundary logic uses IST, not UTC.

many pages for a short window
The API ignores date filters. Even a 2-hour sync may paginate through 10+ pages because the API returns all records newest-first and the code must scan until it passes the start boundary.
.env changes require two restarts
The web server (gunicorn) and the daemon (sync_calls --daemon) are separate processes. Both read .env once at startup. After any .env change, restart both.
missing agent data
If an agent's TL/pool shows blank on the dashboard, the most likely cause is a mismatch between the email prefix the API sends (e.g. irfan from irfan@bluewave.com) and what's in the xlsx. Check Sheet2 "Dialer Name" column for that exact prefix.

Sync commands

# one-shot — last 2 hours from all portals
python manage.py sync_calls

# live daemon — polls every 4 min, fetches 9 min each time
python manage.py sync_calls --daemon --interval 4

# backfill a date range
python manage.py sync_calls --start 2026-05-01 --end 2026-05-09

# flush all records (no sync)
python manage.py sync_calls --flush

# flush then backfill
python manage.py sync_calls --flush --start 2026-05-01 --end 2026-05-09

# import from CSV export
python manage.py sync_calls --csv /path/to/report.csv

After editing Team_tradex.xlsx — both are required

# 1. fix the DB (rewrites all CallRecord rows)
python manage.py reenrich_teams

# 2. fix the in-memory cache in the web process
curl -X POST http://localhost:8000/api/reload-team/

After changing .env

# restart both processes — they each read .env once at startup
sudo systemctl restart stringeex-web
sudo systemctl restart stringeex-sync
