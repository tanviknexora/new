StringeeX — Database inspection, API field reference, and data debugging guide

Database schema — calls_callrecord table

ColumnTypeWhat it means
id PKvarchar(128)StringeeX's own call ID — primary key, never duplicated
account_namevarcharAgent email from API (e.g. anushka@bluewave.com)
directionvarchar'inbound', 'outbound', or 'internal'. API sends 0/1 — mapped at ingest.
start_time_msbigintCall start — Unix milliseconds UTC. Divide by 1000 for seconds.
end_time_msbigintCall end — Unix milliseconds UTC
durationintegerTotal call duration in seconds (ringing + talk)
talk_durationintegerAgent talk time in seconds. If 0 → call was not connected.
answer_durationfloatSame as talk_duration for connected calls (from API)
end_call_reasonvarcharWhy call ended. 'CAN_NOT_MAKE_CALL' rows are skipped at ingest.
customer_numbervarcharExternal phone number dialled / calling in
stringee_numbervarcharThe virtual Stringee number used
full_name teamvarcharAgent display name from Team_tradex.xlsx
employee_code teamvarcharHR employee code from Excel
pool teamvarcharTeam pool (e.g. Sales, Support) from Excel
tl teamvarcharTeam Leader name from Excel
vertical teamvarcharBusiness vertical from Excel
fetched_attimestamptzWhen WE inserted this record into our DB (auto-set)

Computed properties (not stored): queue_duration_secs = duration − talk_duration  ·  is_connected = talk_duration > 0

See the 10 most recent calls ingested

SELECT id, account_name, direction, start_time_ms, talk_duration, end_call_reason FROM calls_callrecord ORDER BY start_time_ms DESC LIMIT 10;
psql

Readable timestamps — convert ms to local time

SELECT account_name, to_timestamp(start_time_ms/1000) AT TIME ZONE 'Asia/Dubai' AS call_time, talk_duration, direction FROM calls_callrecord ORDER BY start_time_ms DESC LIMIT 20;
psql

Calls today (Dubai time)

SELECT COUNT(*) FROM calls_callrecord WHERE to_timestamp(start_time_ms/1000) AT TIME ZONE 'Asia/Dubai' >= CURRENT_DATE AT TIME ZONE 'Asia/Dubai';
psql

Calls per agent today — ranked by count

SELECT account_name, full_name, tl, COUNT(*) AS total, SUM(CASE WHEN talk_duration > 0 THEN 1 ELSE 0 END) AS connected FROM calls_callrecord WHERE to_timestamp(start_time_ms/1000) AT TIME ZONE 'Asia/Dubai' >= CURRENT_DATE AT TIME ZONE 'Asia/Dubai' GROUP BY account_name, full_name, tl ORDER BY total DESC;
psql

Agents with no team enrichment (null pool/tl) — missing from Excel

SELECT DISTINCT account_name, full_name FROM calls_callrecord WHERE pool IS NULL OR tl IS NULL ORDER BY account_name;
psql

Distribution of end_call_reason values (what's being filtered out)

SELECT end_call_reason, COUNT(*) AS n FROM calls_callrecord GROUP BY end_call_reason ORDER BY n DESC;
psql

Date range of data in the DB — oldest and newest call

SELECT to_timestamp(MIN(start_time_ms)/1000) AT TIME ZONE 'Asia/Dubai' AS oldest, to_timestamp(MAX(start_time_ms)/1000) AT TIME ZONE 'Asia/Dubai' AS newest FROM calls_callrecord;
psql

Calls per day — data volume overview

SELECT DATE(to_timestamp(start_time_ms/1000) AT TIME ZONE 'Asia/Dubai') AS day, COUNT(*) AS calls FROM calls_callrecord GROUP BY day ORDER BY day DESC LIMIT 14;
psql

Check when records were last fetched (sync health)

SELECT MAX(fetched_at) AS last_sync, COUNT(*) FILTER (WHERE fetched_at > NOW() - INTERVAL '10 minutes') AS synced_last_10min FROM calls_callrecord;
psql

Look up all calls from a specific agent email

SELECT to_timestamp(start_time_ms/1000) AT TIME ZONE 'Asia/Dubai' AS time, direction, talk_duration, end_call_reason, customer_number FROM calls_callrecord WHERE account_name ILIKE '%anushka%' ORDER BY start_time_ms DESC LIMIT 20;

Replace anushka with any name or email fragment

API response field decoder

/api/stats/

What each field means in the KPI response

total all calls in the period (including unanswered)
connected calls where talk_duration > 0 (agent actually spoke)
connect_rate connected ÷ total × 100, rounded to 1dp
inbound / outbound direction counts
avg_talk_secs average talk time in seconds (connected calls only)
total_talk_mins total agent talk time in minutes
total_dur_mins total duration including ringing/queue in minutes
/api/calls/

Key fields in each call row

start_time_ms Unix ms UTC — divide by 1000 to get epoch seconds
duration total seconds (ringing + queue + talk)
talk_duration seconds the agent was actually talking (0 = not answered)
queue_duration_secs duration − talk_duration (time before agent picked up)
talk_duration_display / queue_duration_display formatted as HH:MM:SS
is_connected true if talk_duration > 0
end_call_reason terminal state from StringeeX API
/api/agent-summary/

Per-agent rollup fields

total_calls all calls assigned to this agent
connected_calls calls where talk_duration > 0
connect_rate percentage (0–100)
talk_time total talk time in HH:MM:SS
queue_time total time in queue in HH:MM:SS
total_time sum of all call durations in HH:MM:SS
/api/hourly-breakdown/

Understanding the nested hourly structure

hours sorted list of hours with data (e.g. [9, 10, 11, 12])
rows[].hourly[H].calls calls in that hour for that agent
rows[].hourly[H].unique_dials distinct customer numbers in that hour
rows[].totals day total across all hours for this agent
hour_totals[H] grand total across all agents for that hour
hours are IST/Dubai local (not UTC) — 9 means 9am Dubai time

Common gotchas & debugging tips

gotcha

Agent shows in API but has no team data (pool/tl = "—")

Their email prefix isn't in Sheet2 of Team_tradex.xlsx. Add them there and run python manage.py reenrich_teams, then POST /api/reload-team/.

gotcha

Dashboard shows zero calls for today but the DB has data

The date filter uses Dubai/IST midnight boundaries. Check that start_time_ms falls within the correct UTC window — run the readable timestamp query above to verify.

gotcha

talk_duration is 0 for all calls — nothing is "connected"

Likely a CSV import where the answer_duration column is null/empty for all rows. Check the raw CSV. Also verify end_call_reason distribution — if most are 'CAN_NOT_MAKE_CALL' they were filtered; if 'no_answer' they genuinely weren't picked up.

gotcha

Duplicate-looking records after a re-sync

Can't happen — id is the primary key. If today's record looks stale, the daemon uses update_or_create on today's data. Verify fetched_at of the latest record vs current time.

gotcha

Hourly breakdown shows calls in wrong hour (off by timezone)

Hours are bucketed in IST (UTC+5:30), same as Dubai. A call at 3:30am UTC appears in hour 9. If hours look shifted by ±4–5, the timezone config may have changed. Check LOCAL_TZ in views.py.
