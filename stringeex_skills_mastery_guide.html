StringeeX — Skills and knowledge areas to master this project

1 — Team data management (Excel)

Excel

Understanding Team_tradex.xlsx structure

essential

Every agent's name, TL, pool, and vertical comes from this file. If it's wrong, everything downstream is wrong — the dashboard, the API, the DB.

  • Sheet1 = full CRE list keyed on Email (nexora.live addresses). Columns: Email, Employee Code, Full Name, Pool, TL, Vertical.
  • Sheet2 = Stringee-dialer agents. Columns: Dialer, Dialer Name, Email, Employee code, Full Name, Pool, TL, Vertical. Only rows where Dialer = "stringee" (or "strigee") are loaded.
  • Sheet2 overrides Sheet1 for the same agent prefix. If a person is in both, Sheet2 wins.
  • Matching is by email prefix only — everything before the @. So "irfan@bluewave.com" and "irfan@nexora.live" both match the key "irfan".
  • If an agent's pool/TL shows "—" in the dashboard, their row is missing from Sheet2 (or the Dialer column isn't "stringee").
  • After editing the file: run python manage.py reenrich_teams then POST /api/reload-team/.
Excel

Adding a new agent correctly

essential
  • Add them to Sheet2 with Dialer = "stringee" and Dialer Name = their bluewave.com API email.
  • Their nexora.live email goes in the Email column (used as fallback reference only).
  • If the prefix already exists in Sheet1, Sheet2 values will override — that's intentional.
  • If a field in Sheet2 is blank, the code falls back to Sheet1's value for that field.

2 — StringeeX API behaviour

API

How the API actually works (critical quirk)

essential

The StringeeX API ignores its own date filter parameters. Knowing this explains why sync_calls works the way it does.

  • The API always returns latest records DESC regardless of start_time / end_time params sent.
  • The code works around this by paginating until records go older than start boundary, then stops — collecting only records within the desired window.
  • page_size=1000 per page. Large backfills = many API pages = slow. That's expected, not a bug.
  • Multiple portals (bluewave2, bluewave3) are fetched separately and merged. Cross-portal duplicates are deduplicated by call id.
  • Auth token is fetched fresh each sync run via POST /v1/account — it's not cached between runs.
  • If a portal is unreachable, its error is logged but other portals still run — partial sync rather than full failure.
API

What the raw API fields mean before mapping

useful
  • direction from API is an integer: 0 = outbound, 1 = inbound. The code maps this at ingest — what's in the DB is already the string.
  • start_time from API = Unix milliseconds UTC. Stored as-is in start_time_ms.
  • answer_duration from API is a float in seconds, can be null. Stored in answer_duration FloatField.
  • end_call_reason = 'CAN_NOT_MAKE_CALL' records are skipped entirely at ingest — they never reach the DB.
  • The field account_name in the API was previously called account — the current code uses the new name.
  • _portal is a field the code adds itself (not from API) to track which portal a record came from.

3 — Sync and upsert logic

Python

Smart upsert — today vs history behaviour

essential

Understanding this prevents confusion when re-syncing and wondering why some records update and others don't.

  • Today's records use update_or_create — if the call already exists in DB it gets refreshed. Live data stays current.
  • Historical records (any day before today in IST) use get_or_create — existing records are never overwritten.
  • "Today" is determined in IST/Asia:Kolkata time (UTC+5:30), same as Dubai. Midnight IST = when records switch from "today" to "history".
  • The --flush flag wipes the entire calls_callrecord table. Always follow with a backfill.
  • Running sync_calls --csv uses the same upsert logic — today's CSV rows update existing records.
Python

Daemon interval and overlap strategy

useful
  • The daemon runs every N minutes and fetches the last 9 minutes (not just N) — the overlap prevents gaps if a sync runs slightly late.
  • Since today's records use update_or_create, overlapping fetches update records rather than duplicating them.
  • Recommended interval is 4 minutes (set in .env as POLL_INTERVAL_MINUTES=4).
  • If the daemon crashes and restarts, it fetches the last 9 minutes — so short outages <9 min self-heal automatically.

4 — Access control and token system

Auth

How the dashboard token gate works

essential

You need to understand this to share access safely and to revoke it when needed.

  • Set ACCESS_TOKEN in .env. Share this URL once: https://yourdomain.com/?token=<value>.
  • On first visit the token is validated and stored in the user's browser session. The URL param disappears — future visits to the root URL work without the token in the URL.
  • Sessions last 2 weeks (SESSION_COOKIE_AGE in settings.py).
  • To revoke access: change ACCESS_TOKEN in .env and restart the server. All sessions invalidate immediately.
  • Leave ACCESS_TOKEN blank in .env to disable protection entirely (dev only).
  • The middleware is in calls/auth_middleware.py — the API endpoints are also protected by it, not just the dashboard.
  • Django admin (/admin/) uses its own Django user auth — separate from the token system.

5 — Django admin panel

Django

Using /admin/ as a visual database browser

useful

The fastest way to inspect, filter, and manually edit records without writing SQL or Python.

  • Go to http://localhost:8000/admin/ — log in with the superuser you created.
  • Shows all CallRecord rows with filtering, search, and pagination.
  • You can filter by direction, date, and any field exposed in calls/admin.py.
  • You can manually edit or delete individual records — useful for fixing bad test data.
  • The list_display, list_filter, and search_fields in admin.py control what's visible and searchable.
  • If you extend the model (new field), add it to admin.py so it appears in the panel.

6 — .env configuration mastery

Config

Every .env variable and what it controls

essential
  • DJANGO_SECRET_KEY — never share this. If leaked, rotate it (all sessions invalidate).
  • DEBUG=False — always False in production. True makes Django show full tracebacks in browser (a security risk).
  • ALLOWED_HOSTS — comma-separated. Must include your server IP/domain or Django will reject requests with 400 Bad Request.
  • DB_NAME / DB_USER / DB_PASSWORD / DB_HOST / DB_PORT — PostgreSQL connection. DB_NAME=cdr, DB_USER=admin in this project.
  • STRINGEEX_EMAIL / STRINGEEX_PASSWORD — StringeeX portal login. Used to fetch an auth token on every sync run.
  • STRINGEEX_PORTALS — comma-separated portal subdomains (e.g. bluewave2,bluewave3). Takes priority over STRINGEEX_PORTAL.
  • STRINGEEX_PORTAL — single-portal fallback (only used if PORTALS is empty).
  • POLL_INTERVAL_MINUTES — daemon polling interval in minutes. Default 4.
  • ACCESS_TOKEN — dashboard access token. Blank = no protection.
  • After editing .env you must restart the server and daemon — the file is read at startup only.

7 — Timezone handling

Data

How time zones flow through the whole stack

essential

Time zone bugs are the #1 cause of "why does today show 0 calls?" and "why are hours off by 4?"

  • DB stores raw UTC milliseconds — start_time_ms is always UTC. No conversion happens at write time.
  • All date boundaries use IST (Asia/Kolkata, UTC+5:30) — which is the same as Dubai time for business purposes. Midnight IST = midnight Dubai.
  • When you query /api/stats/?date=2026-05-13, the backend converts that date to IST midnight → IST 23:59:59 → UTC ms, then filters.
  • Hourly breakdown hours are IST local hours — hour 9 means 9am Dubai. The _ms_to_local_hour() function does this conversion.
  • In PostgreSQL, convert ms to readable time with: to_timestamp(start_time_ms/1000) AT TIME ZONE 'Asia/Dubai'.
  • TIME_ZONE = 'Asia/Kolkata' in settings.py — this affects Django's auto_now_add fields like fetched_at.
  • If you ever change the timezone in settings.py, existing ms timestamps are unaffected (they're UTC) but display logic and date filters will shift.

8 — PostgreSQL skills for this project

Postgres

psql meta-commands worth knowing

useful
  • \dt — list all tables in the database
  • \d calls_callrecord — describe the table (columns, types, constraints)
  • \di calls_callrecord — list indexes on the table
  • \x — toggle expanded output mode (one column per line — great for wide rows)
  • \timing — show query execution time
  • \q — quit psql
  • \l — list all databases on the server
  • \c cdr — connect/switch to the cdr database
Postgres

Performance and index awareness

advanced
  • The table's primary key is id (varchar). Django auto-indexes it.
  • Most queries filter on start_time_ms — if the table grows large, add an index: CREATE INDEX ON calls_callrecord (start_time_ms DESC);
  • Queries filtering by account_name or tl also benefit from indexes as data grows.
  • Check query plans with: EXPLAIN ANALYZE SELECT ... FROM calls_callrecord WHERE ...;
  • Table size check: SELECT pg_size_pretty(pg_total_relation_size('calls_callrecord'));

9 — Extending and modifying the project

Django

Adding a new field to the model

advanced
  • Edit calls/models.py — add the field definition.
  • Run python manage.py makemigrations — generates the migration file.
  • Run python manage.py migrate — applies it to the DB.
  • Update sync_calls.py to populate the field during ingest.
  • Update calls/admin.py to show it in the admin panel.
  • Update the relevant view in views.py to include it in the API response.
Django

Adding a new API endpoint

advanced
  • Add a view function to calls/views.py — decorate with @require_GET and return a JsonResponse.
  • Add a path() entry to calls/urls.py.
  • Use _day_ms_range(date_str) for date filtering — it handles the IST boundary conversion for you.
  • The auth middleware automatically protects all routes — no extra code needed.
API

Adding a new portal

essential
  • In .env, add the portal subdomain to STRINGEEX_PORTALS: e.g. STRINGEEX_PORTALS=bluewave2,bluewave3,bluewave4
  • Restart the server and daemon — that's all. The code loops over all portals automatically.
  • Cross-portal duplicate calls are deduplicated by id automatically.
  • If one portal fails auth, the rest still sync — check logs for per-portal errors.

10 — Daily operations runbook

Ops

Morning health check (30 seconds)

essential
  • Check daemon is running: sudo systemctl status stringeex-sync — should say "active (running)".
  • Check last sync time: curl -s http://localhost:8000/api/last-sync/ | python -m json.tool
  • Check today's call count: curl -s "http://localhost:8000/api/stats/?date=$(date +%Y-%m-%d)" | python -m json.tool
  • If daemon is down: sudo systemctl restart stringeex-sync then check journalctl -u stringeex-sync -n 50 for the error.
Ops

After updating Team_tradex.xlsx (workflow)

essential
  • Replace data/Team_tradex.xlsx on the server.
  • Re-enrich all existing DB records: python manage.py reenrich_teams
  • Hot-reload the in-memory lookup without restart: curl -X POST http://localhost:8000/api/reload-team/
  • Verify in dashboard that affected agents now show correct pool/TL.
Ops

Monthly data reset + backfill workflow

useful
  • Stop the daemon: sudo systemctl stop stringeex-sync
  • Flush all data: python manage.py sync_calls --flush
  • Backfill from desired start date: python manage.py sync_calls --start 2026-05-01 --end 2026-05-31
  • Restart daemon: sudo systemctl start stringeex-sync
  • Verify record count in DB matches expectations.
